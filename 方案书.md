# 邻里童享项目方案书

更新日期：2026-04-22

## 一、项目概述

### 1. 项目名称
邻里童享

### 2. 项目定位
面向 3-5 公里邻里社区范围内家庭的儿童闲置物品交换与转让公益平台，聚焦玩具、绘本、童书、童车、安全座椅、餐椅、积木等儿童成长型物品的高频流转场景。

### 3. 核心价值
- 区别于综合二手平台，强调“同小区及周边邻里”小范围可信流转。
- 聚焦儿童闲置，降低筛选成本，提升匹配效率。
- 兼顾公益感、社区温度和正规可信度，适合从熟人圈、邻居圈冷启动。
- 支持演示期快速验证项目可行性，再逐步升级为正式商业化运营版本。

## 二、立项背景与市场机会

### 1. 真实痛点
- 儿童用品更换频率高，很多家庭短期内就会产生大量闲置。
- 闲鱼等平台虽然流量大，但筛选成本高、信任弱、沟通距离远。
- 家长更愿意在熟悉小区和周边范围内完成面交、自提、交换。
- 社区内部往往已有需求，但缺少一个足够轻、足够有温度、又足够正规的平台承接。

### 2. 本项目的切入点
- 以“小区 + 周边 3 公里”为默认流转范围，降低交易半径。
- 用“儿童闲置”垂直品类提高平台认知度和目标用户匹配效率。
- 用“公益流转 + 邻里互助 + 温馨设计”强化传播与口碑。
- 先用 Demo 版本低成本验证用户活跃度、复购率、发布率和推荐意愿，再决定是否扩大投入。

## 三、项目目标

### 1. 第一阶段目标
完成一个可被真实用户体验、可被老板审批、可进行小范围试运营的高保真 Demo 平台。

### 2. 本阶段成功标准
- 用户可通过 QQ 邮箱验证码完成注册/登录。
- 用户可选择小区，查看同小区及周边 3 公里内的物品，并支持扩展到 5 公里。
- 用户可发布真实图文物品信息，图片可正常上传与展示。
- 用户可查看详情、发起沟通、浏览“我的”页面。
- 平台具备基础运营配置能力与演示数据，整体体验不再是“纯壳子”。
- 网站具备移动端优先体验，同时兼顾电脑端浏览。

## 四、当前版本完成情况

截至 2026-04-22，当前项目已经完成以下能力。

### 1. 前台用户端
- 首页推荐流可展示真实 Demo 物品数据。
- 响应式 H5 页面已支持手机与电脑端访问。
- 默认按用户小区及周边 3 公里范围展示，支持扩展到 5 公里。
- 支持物品详情页查看、分享页跳转和基础图文分享 meta。
- 支持邮箱验证码登录/注册一体化流程。
- 支持发布物品、上传图片、填写类目、年龄段、交换方式、成色、标签等信息。
- 支持消息页、我的主页、我的发布、徽章页、设置页等基础页面。

### 2. 运营与后台能力
- 已接入飞书多维表格，用于小区配置、运营配置、发布审核队列。
- 已具备运营配置回退能力：飞书异常时可回落到本地默认配置，不影响 Demo 基础可用性。
- 已具备演示数据种子能力，可快速形成“平台已经开始运营”的初始观感。

### 3. 基础集成能力
- 已接入 QQ 邮箱 SMTP 真发信链路，用于发送登录验证码。
- 已接入腾讯云 COS 作为图片上传存储链路。
- 已对 COS 私有资源增加签名读取逻辑，图片详情展示链路已恢复可用。
- 已接入腾讯地图能力，用于小区定位、搜索和附近范围能力。

### 4. 设计与体验
- 页面已按“温馨童趣、简约轻奢、社区公益感”方向做过多轮优化。
- 移动端导航、首页结构、发布页、分享页、顶部栏、筛选栏已做针对性修正。
- 去除了部分不适合直接出现在用户页面的内部思考式文案。

## 五、当前系统架构

### 1. 技术架构
- 前端：Next.js 16 + React 19 + Tailwind 风格化样式体系
- 后端：Node.js + Express
- 业务数据库：本地 SQLite（`sql.js` 持久化到 `server/data/swap.db`）
- 邮件：QQ 邮箱 SMTP
- 图片存储：腾讯云 COS
- 地图与定位：腾讯地图
- 运营配置与审核：飞书多维表格

### 2. 当前数据分层
- 用户、物品、消息、交换、通知等核心业务数据：当前保存在本地 SQLite
- 小区列表、运营配置、审核队列：已支持飞书多维表格
- 图片资源：腾讯云 COS

### 3. 当前部署形态
- 前端和后端均可本地生产模式启动。
- 当前适合 Demo 演示、真机联调、小范围试运营。
- 若要公网正式发布，还需要完成域名、HTTPS、服务器部署、日志监控、数据备份等正式化收尾。

## 六、当前版本的优势

### 1. 冷启动友好
- 项目已具备完整用户路径，不是单纯页面样板。
- 平台已有多条真实感较强的 Demo 物品，适合给邻居和朋友直接体验。
- 小区和附近范围逻辑强化了“这是邻里平台，不是泛二手平台”的认知。

### 2. 成本低、迭代快
- 前端可部署到 Cloudflare Pages，成本低。
- 邮箱、飞书、多维表格、COS 均可从低成本或免费组合开始。
- 适合先验证“有无需求”和“用户是否愿意发布/交换”，再投入更重的商业化研发。

### 3. 社区传播性强
- 儿童闲置具有天然的熟人推荐属性和分享属性。
- 平台定位明确，传播口径清晰，容易被家长群、小区群理解和接受。

## 七、当前版本的客观边界

这一部分建议老板重点关注。

### 1. 当前已达到
- 可演示
- 可试运营
- 可邀请小范围种子用户体验
- 可作为内部立项评审和外部冷启动验证样板

### 2. 当前尚不建议直接定义为“长期公网正式版”
原因如下：

- 核心业务数据当前仍是本地 SQLite，适合 Demo 与小规模试运营，不适合长期高并发公网运营。
- 缺少正式运维体系，例如日志报警、自动备份、监控看板、异常告警。
- 缺少内容风控、用户申诉、违规处理、管理后台等更完整的正式运营能力。
- 微信外部分享图文预览要稳定生效，仍需要公网域名和正式 HTTPS 环境。

### 3. 建议的表述方式
建议将当前版本定义为：

“可试运营的高保真 Demo + 小范围种子用户验证版本”

这样的定位最真实，也最有利于老板审批资源时做出准确判断。

## 八、正式上线建议方案

### 方案目标
在现有基础上，用尽量低成本的方式完成“可对外正式开放”的第一版。

### 推荐方案

#### 1. 前端部署
- 使用 Cloudflare Pages 部署前端站点
- 绑定正式域名
- 开启 HTTPS

#### 2. 后端部署
- 使用腾讯云轻量服务器或阿里云 ECS 部署 Express API
- 使用 PM2 或同类进程守护方式维持后端稳定运行
- 使用 Nginx/Caddy 做反向代理

#### 3. 数据方案
- 过渡期：保留飞书多维表格作为运营配置和审核后台
- 正式版建议：核心业务数据迁移到云数据库
- 推荐方向：PostgreSQL 或 MySQL
- 飞书多维表格继续承担轻运营、内容配置、社区配置、审核流管理

#### 4. 图片与静态资源
- 继续使用腾讯云 COS
- 配合正式公网域名或 CDN，提升访问稳定性与图片加载速度

#### 5. 邮件与通知
- 初期继续使用 QQ 邮箱 SMTP
- 用户量扩大后再切换为企业级邮件服务

## 九、建议的分阶段推进路线

### 阶段一：当前版本试运营
目标：验证需求是否成立。

建议动作：
- 邀请 20-50 位周边家长体验
- 观察发布率、浏览率、沟通率、交换达成率
- 收集反馈：是否愿意继续使用、最常用类目、最常见疑问

### 阶段二：正式上线收尾
目标：达到可公网开放的标准。

建议动作：
- 换正式公网域名和 HTTPS
- 核心数据迁移到云数据库
- 增加日志监控、自动备份、错误告警
- 补充后台管理与审核机制
- 规范内容协议、用户协议、隐私政策页面

### 阶段三：增长验证
目标：判断项目是否值得继续投入。

重点观察：
- 7 日留存
- 发布转化率
- 交换达成率
- 用户主动推荐率
- 不同小区之间的扩散速度

## 十、预算与成本建议

### 1. 当前阶段
可按低成本试运营方式控制投入。

预计成本项：
- 域名
- 云服务器
- COS 图片存储与流量
- 微信认证或相关公众号配置
- 可能的设计与运营物料成本

### 2. 初期推荐原则
- 先验证可行性，再放大研发和市场投入
- 优先把预算花在“真实用户体验”和“正式上线稳定性”上
- 不建议在需求尚未验证前过早投入重后台和复杂商业功能

## 十一、风险与应对

### 1. 用户活跃不足
风险：
平台冷启动后可能浏览多、发布少。

应对：
- 先从熟人小区和邻里社群导入首批真实发布
- 用 Demo 物品先填充观感，但逐步过渡到真实住户发布

### 2. 平台信任度不足
风险：
新平台天然缺乏交易信任。

应对：
- 强调同小区和周边面交
- 用小区、身份、邻里公益定位建立信任
- 后续补实名认证、发布审核、信用体系

### 3. 正式运维能力不足
风险：
小流量可运行，但正式开放后容易暴露稳定性问题。

应对：
- 正式版前完成日志、监控、备份、容灾最小闭环

### 4. 数据方案扩展性不足
风险：
本地 SQLite 方案不适合长期正式运营。

应对：
- 在老板批准后尽快进入云数据库迁移阶段

## 十二、建议老板审批的事项

建议审批分两步走。

### 第一步：批准试运营
目标：
允许用当前版本在周边小区做 2-4 周种子用户验证。

需要支持：
- 正式域名
- 云服务器
- 基础运维部署
- 少量运营素材与推广动作

### 第二步：视数据决定是否追加正式版预算
重点看以下数据再决定是否继续投入：
- 注册人数
- 发布人数
- 有效物品数量
- 私信沟通量
- 交换/转让达成量
- 用户主动推荐比例

## 十三、结论

邻里童享当前已经不是一个停留在概念层的产品草图，而是一个可以被真实用户体验、可以进入种子用户测试、可以支持老板评估商业可行性的高保真 Demo 平台。

从产品方向看，它切中了“儿童闲置高频流转 + 邻里强信任场景”的明确需求。

从实施成本看，它适合用较低预算快速验证，不必一开始就投入重资产。

从当前状态看，它已经具备试运营条件，但若定义为长期正式公网产品，仍建议在数据层、运维层和合规层再做一次正式化收尾。

综合建议：

建议老板批准本项目进入“小范围试运营验证阶段”，并同步批准正式上线所需的最小资源预算。若试运营数据达标，再进入正式版加速建设。
